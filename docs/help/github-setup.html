SYSEN 5470 · Help · GitHub Setup

Clone, Commit,
Push.

From "I've never used Git" to (a) a local copy of the course repo and (b) your own private repo for the project case studies. Plan on 30 minutes the first time, two minutes per week after that. If Git is already in your toolkit, skip to step 4.

Six steps
  1. 1Install Git
  2. 2Make a GitHub account
  3. 3Clone the course repo
  4. 4Make your project repo
  5. 5Basic add / commit / push
  6. 6Troubleshooting

1.Install Git

Git is the command-line tool. GitHub is the hosting service. You need Git locally before GitHub becomes useful.

  • macOS: open Terminal and run git --version. If it prompts you to install Xcode command-line tools, accept. Otherwise you already have it.
  • Windows: install Git for Windows. Includes Git Bash, which gives you a Unix-style terminal — keep this checked.
  • Linux: sudo apt install git (Debian/Ubuntu) or your distro's equivalent.

Verify:

git --version # → git version 2.43.0 (any 2.x is fine)

Tell Git who you are. Do this once per machine:

git config --global user.name "Your Name" git config --global user.email "you@example.com" git config --global init.defaultBranch main

2.Make a GitHub account

If you already have one, skip. If not:

  • Sign up at github.com/signup. The free tier is plenty for this course.
  • Use github.com, not a GitHub Enterprise instance (like github.coecis.cornell.edu). The course tools assume public github.com.
  • Use the same email you'll use with Git in step 1, or add it as a secondary email under Settings → Emails.
  • Pick a username that's recognizably you (real name or close to it). Customize your profile (name + photo) so classmates and Tim can find you.

Generate a Personal Access Token (PAT)

You'll need a PAT to push code from the command line. Go to Settings → Developer settings → Personal access tokens. You'll see two options:

  • Classic token — covers all your repos. Quick. Pick the repo scope (and public_repo if you'll work with public repos too).
  • Fine-grained token — scoped to specific repositories. More surgical. Under Permissions → Repository → Contents set Read and Write. This auto-adds the Metadata permission.

Either works for the course. Pick an expiration date after July 13, 2026. Save the token in your password manager the moment you see it — GitHub only shows it once.

When you push code for the first time, Git will prompt for a username and password. Use your GitHub username and paste the PAT as the password. Most systems then cache the credential.

SSH alternative: if you'd rather use SSH keys than PATs, follow GitHub's SSH key guide. The course doesn't care which you pick.

3.Clone the course repo

The course materials live at github.com/timothyfraser/netsci. Cloning makes a local copy you can update with git pull.

Pick a folder where you keep code (commonly ~/code/ or ~/projects/) and run:

# navigate to your code folder, then: git clone https://github.com/timothyfraser/netsci.git cd netsci ls

You should see folders like docs/, code/, and a top-level README.md. To pull the latest updates each week:

cd netsci git pull
You won't push to this repo — it's read-only for students. The course pushes to you; you push your own work to a separate repo (step 4).

4.Make your project repo

Your project case studies live in your own private GitHub repo. Setup, in order:

  1. On GitHub, click New repository. Name it something like netsci-project. Set it to Private. Don't bother adding a README from the web UI — we'll create one locally.
  2. On your laptop, in a folder you control:
# initialize and create the basic structure mkdir netsci-project cd netsci-project git init mkdir code data report # a starter README echo "# My SYSEN 5470 Project" > README.md echo "3 project case studies. Each lives under code/." >> README.md # first commit git add . git commit -m "Initial commit" # connect to your empty GitHub repo (replace USERNAME) git remote add origin https://github.com/USERNAME/netsci-project.git git branch -M main git push -u origin main

Now your project repo exists on GitHub.

Add Tim as a collaborator (important)

Since the repo is private, Tim can't see it until you invite him:

  1. Open your repo on GitHub → Settings → Collaborators.
  2. Click Add people, type timothyfraser, send the invite.
  3. You're done — Tim doesn't need to accept before you submit. Take a screenshot of the Collaborators page for your records.

Suggested layout

netsci-project/ ├── README.md # short overview + which 3 case studies you picked ├── code/ │ ├── 01-centrality/ # your first project case study │ │ ├── project.R # OR project.py │ │ └── report.pdf │ ├── 02-clustering/ │ └── 03-routing/ ├── data/ # raw inputs (CSVs, edgelists) └── notes/ # scratch, sketches, anything

5.Basic add / commit / push

The minimum workflow you need to keep your work safe. After every meaningful chunk of work:

# 1. see what's changed git status # 2. stage the changes you want to commit git add code/01-centrality/project.R # or stage everything modified: git add . # 3. record the change with a short message git commit -m "Centrality: first pass, top-10 nodes by betweenness" # 4. push to GitHub git push

That's the whole loop. Commit often, in small chunks, with short messages. Commit messages don't need to be poetic — "wip" and "fix off-by-one in degree calc" are both fine. The point is to leave breadcrumbs so a future you (or Tim reviewing) can trace what changed and when.

Pull other people's changes before you push

git pull # pull down anything new (rare on your own private repo, but a good habit) git push # then send yours up
Heads-up — what NOT to commit: large datasets (over ~50MB), credentials (.env files, API keys), or anything from inside a node_modules/ or .Rproj.user/ directory. Add a .gitignore to the repo with the common patterns (github/gitignore has templates).

Same loop from inside R / Positron

If you'd rather not leave your IDE, three R packages handle Git for you. Install once:

install.packages(c("usethis", "gert", "credentials"))

Then, once per machine, hand R your PAT and protect against accidental commits of secrets:

library(usethis) library(gert) library(credentials) # Store your PAT in the OS keychain so R can find it credentials::set_github_pat() # Globally vaccinate against committing .env, .Rhistory, .DS_Store, etc. usethis::git_vaccinate()

After that, the per-commit loop is three lines:

gert::git_pull() gert::git_add(dir(all.files = TRUE)) gert::git_commit_all("Centrality: first pass") gert::git_push()
From Python: there's no Pythonic equivalent that's worth recommending — open a terminal in Positron / VS Code and run the git commands above. That's what most working scientists do regardless of language.

6.Troubleshooting

fatal: not a git repository

You're running Git from a folder that hasn't been initialized. cd into the folder that has the .git directory, or run git init if you intended to start one here.

Please enter the commit message… stuck in vim

Press Esc, type :wq, press Enter. Or use git commit -m "your message" next time to skip the editor entirely.

Updates were rejected because the tip of your branch is behind

Someone (probably another machine you also use) pushed before you did.

git pull --rebase git push

Permission denied (publickey)

You're using SSH but your key isn't registered with GitHub. Either set up the SSH key (GitHub's SSH guide) or switch the remote to HTTPS:

git remote set-url origin https://github.com/USERNAME/netsci-project.git

I accidentally committed a big file / a secret

If you haven't pushed yet:

git reset HEAD~ # undo the last commit, keep the file changes # then add the file to .gitignore and recommit

If you already pushed a secret: rotate it. Git history doesn't forgive — the secret is in the repo's history even if you delete the file in a new commit. Treat it as compromised, generate a new one, and update the upstream service.
